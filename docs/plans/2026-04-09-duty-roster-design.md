# 值班展示系统设计说明

**日期：** 2026-04-09

## 1. 项目目标

建设一个移动端优先、兼容 PC 端的值班展示系统。系统主要服务于日常查看值班安排的普通用户，不引入复杂权限体系；仅提供一个独立的管理员上传页面，用于后续导入 Excel 数据。

当前已确认的业务目标如下：

- 首页按日期展示各单位值班摘要。
- 顶部日期区域默认折叠，减少对列表区域的占用。
- 每个单位首页只展示三类固定信息：值班领导、值班中层、值班人员。
- 点击单位名称后进入详情页，查看该单位当天的完整联系人列表。
- 详情页核心字段为部门名称、姓名、电话；当前真实 Excel 暂时缺少电话，系统需先按可空字段设计，后续补齐。
- 管理员通过独立页面上传 Excel，上传前需要输入服务端配置的密码。

## 2. 方案选择

针对当前需求，存在三种可选方向：

### 方案 A：纯前端页面 + 本地假数据

优点：

- 页面能最快做出来。
- 适合早期确认视觉和交互。

缺点：

- 后续接上传、密码校验、数据库时需要重构。
- 无法承接正式上线。

### 方案 B：前后端分离

优点：

- 职责清晰，利于中大型项目演进。
- 前后端可以独立开发和部署。

缺点：

- 初期项目拆分较重。
- 对当前“先快速做出第一版”的目标来说成本偏高。

### 方案 C：单体全栈应用

优点：

- 开发与部署成本低。
- 页面、接口、上传逻辑可在同一项目内推进。
- 后续接 Excel 导入和数据库最自然。

缺点：

- 后期若系统复杂度大幅提升，仍可能需要拆分。

**推荐采用方案 C：`Next.js + MySQL（或 PostgreSQL）+ Node.js 服务端部署`。**

推荐理由：

- 兼容“先用模拟数据完成展示，再接真实导入”的迭代方式。
- 能直接承接管理员上传、服务端密码校验和数据库落库。
- 项目规模较轻时，开发和部署成本最合适。

## 3. 页面结构

系统首期包含三个页面。

### 3.1 首页

首页承担“当天值班总览”的职责，整体保持用户提供截图的浏览逻辑。

页面结构建议如下：

- 顶部展示当前日期摘要，例如“2026 年 4 月 9 日”。
- 日期区域默认折叠，仅在用户需要切换日期时展开日历面板。
- 日期切换支持选择日历中的某一天，并支持快捷返回今天。
- 下方按单位顺序展示值班卡片。
- 每张卡片仅展示单位名称和三类固定值班信息。

首页卡片字段固定为：

- 值班领导
- 值班中层
- 值班人员

每个字段建议拆分为姓名和电话两个子字段进行存储；当前电话为空时，前端可显示“待补充”或留空，但布局必须预留电话位置，避免后期补电话时调整 UI 结构。

### 3.2 单位详情页

用户点击首页的单位名称后进入单位详情页。

该页面不再强调岗位名称，而是强调“快速找到联系人”的使用目的。页面结构建议如下：

- 顶部显示返回按钮、单位名称、当前日期。
- 主体区域按部门名称分组展示联系人。
- 每条联系人记录展示：部门名称、姓名、电话。

前端默认按部门分组展示，原因如下：

- 同一部门多人时，可读性更好。
- 手机端滚动浏览更容易定位。
- 后续 Excel 如果存在同部门多人的情况，不需要再改详情页结构。

电话字段后续补齐后，手机端点击号码直接使用 `tel:` 发起拨号；PC 端保持正常显示即可。

### 3.3 管理员上传页

管理员上传页与前台查看页面分离，不放入普通用户的主浏览流程中。

页面建议包含：

- Excel 文件选择区域。
- 密码输入框。
- 上传按钮。
- 上传结果反馈区域。

服务端校验逻辑如下：

- 前端提交文件和密码。
- 服务端从环境变量或服务端配置中读取上传密码。
- 校验通过才允许进入解析或导入流程。
- 校验失败则返回明确错误提示。

## 4. 数据模型

为兼容“首页摘要”和“详情联系人列表”两种展示方式，数据库建议采用以下核心模型。

### 4.1 单位表 `units`

用于存储首页展示单位。

建议字段：

- `id`
- `name`
- `code`
- `sort_order`
- `is_active`
- `created_at`
- `updated_at`

### 4.2 首页摘要表 `duty_overviews`

用于存储某一天、某单位在首页展示的三类固定值班信息。

建议字段：

- `id`
- `duty_date`
- `unit_id`
- `leader_name`
- `leader_phone`
- `middle_manager_name`
- `middle_manager_phone`
- `staff_name`
- `staff_phone`
- `source_batch_id`
- `created_at`
- `updated_at`

### 4.3 详情联系人表 `duty_contacts`

用于存储单位详情页的完整联系人列表。

建议字段：

- `id`
- `duty_date`
- `unit_id`
- `department_name`
- `person_name`
- `phone`
- `sort_order`
- `source_batch_id`
- `created_at`
- `updated_at`

### 4.4 导入批次表 `import_batches`

用于记录每次上传或导入的元信息，便于后续追踪和重导。

建议字段：

- `id`
- `file_name`
- `storage_path`
- `import_status`
- `error_message`
- `imported_at`
- `created_at`
- `updated_at`

## 5. 接口设计

首期接口保持最小化。

### 5.1 前台查询接口

`GET /api/duty/overview?date=YYYY-MM-DD`

返回指定日期下所有单位首页摘要数据。

`GET /api/duty/units/{unitId}/contacts?date=YYYY-MM-DD`

返回指定日期、指定单位的详情联系人数据，并按部门分组返回。

### 5.2 管理端接口

`POST /api/admin/imports`

提交 Excel 文件与上传密码，由服务端统一校验和处理。首期即使真实 Excel 映射尚未完成，也建议先保留该接口结构，后续只补解析逻辑。

## 6. Excel 适配策略

当前已知真实 Excel 暂时缺少电话字段，因此导入逻辑应采用“可空兼容”策略。

首期实现建议：

- 先固定数据库字段，允许 `leader_phone`、`middle_manager_phone`、`staff_phone`、`phone` 为空。
- 前端在缺失电话时显示占位文案或空状态，不阻塞页面使用。
- 待后续拿到完整电话数据后，再补导入映射和手机号校验逻辑。

为降低后期接真实模板的风险，导入层建议单独封装，不要把 Excel 列名直接散落到接口或页面逻辑中。

## 7. 错误处理

首期需要覆盖的错误场景包括：

- 查询日期无数据。
- 管理员密码错误。
- 上传文件格式错误。
- 导入解析失败。
- 电话缺失。

建议处理原则：

- 前台无数据时返回清晰空状态，不显示系统异常文案。
- 管理端错误提示要明确指出原因，但不暴露服务端配置细节。
- 导入接口应记录失败批次，方便复盘。

## 8. 测试范围

首期测试应覆盖以下内容：

- 首页日期折叠与展开交互。
- 首页按日期切换后的数据刷新。
- 首页单位卡片展示是否固定为三项摘要。
- 点击单位后跳转详情页。
- 详情页是否按部门分组展示联系人。
- 电话为空时页面显示是否稳定。
- 上传密码校验成功与失败路径。
- 上传接口的基础文件校验。

## 9. 后续演进

待真实 Excel 模板完全明确后，下一阶段可继续补充：

- Excel 列名与数据库字段的映射规则。
- 导入覆盖策略（按日期覆盖、按批次回滚等）。
- 电话字段完整导入与手机号格式校验。
- 上传历史记录查看。
- 更精细的导入结果提示。

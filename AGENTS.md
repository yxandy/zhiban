## 项目概况

本项目是“值班展示系统”，目标是提供一个移动端优先、兼容 PC 端的网页应用，面向普通用户展示各单位每日值班信息，并为管理员提供一个独立的 Excel 上传入口。

当前确认的技术路线如下：

- 前端与服务端采用单体全栈方案。
- 首选技术栈为 `Next.js + TypeScript + Node.js`。
- 数据库固定采用 `MySQL`。
- ORM 优先采用 `Prisma`。
- 样式方案优先采用 `Tailwind CSS`。

当前运行前提如下：

- 开发机本地只运行 Web 应用。
- 数据库运行在局域网内其他机器上。
- 应用通过环境变量中的 `DATABASE_URL` 连接远程 MySQL。
- 开发阶段仍保留模拟数据模式，避免数据库不可用时阻塞页面开发。

## 业务约定

### 首页

- 首页保留“值班日历 + 单位摘要列表”的主结构。
- 顶部日期区域默认折叠，仅在用户切换日期时展开。
- 首页每个单位固定展示三项摘要：
  - 值班领导
  - 值班中层
  - 值班人员
- 单位名称可点击，跳转到该单位的详情页。

### 详情页

- 详情页展示的是“该单位当天的完整联系人列表”，不是下级单位树。
- 详情页主字段为：
  - 部门名称
  - 姓名
  - 电话
- 详情页优先按“部门名称”分组展示。
- 手机端如有电话则支持点击后直接拨打；如无电话则显示“待补充”或等效空状态。

### 上传页

- 管理员上传页面独立存在，不纳入普通用户主浏览流程。
- 上传前需要输入密码。
- 密码必须在服务端配置，不得硬编码到前端。
- 在拿到最终 Excel 模板前，可以先实现上传页、密码校验和占位导入流程。

## 数据设计约定

系统数据至少拆分为以下几类：

- `units`：单位基础信息。
- `duty_overviews`：首页三项摘要数据。
- `duty_contacts`：详情联系人列表数据。
- `import_batches`：导入批次记录。

当前已知真实 Excel 暂时缺少电话字段，因此以下字段必须按“可空”设计：

- 首页摘要中的电话字段。
- 详情联系人中的电话字段。

后续拿到真实 Excel 模板后，应通过独立的导入层做字段映射，不要把 Excel 列名直接耦合到页面组件或接口返回中。

## 开发约定

- 实现顺序优先为：页面原型 -> 模拟数据 -> 服务端接口 -> 数据库接入 -> Excel 导入。
- 在没有真实 Excel 映射前，允许使用模拟数据推进首页、详情页和上传页交互。
- 移动端优先设计，但必须同时检查 PC 端常见宽度下的展示稳定性。
- 所有与时间、日期相关的展示，优先使用中国用户易理解的格式。
- 非必要不要引入权限体系；当前只有上传密码校验这一项后台保护措施。
- 本地开发默认优先兼容“连接局域网远程 MySQL”场景，不假设数据库与 Web 应用部署在同一台机器。

## Git 约定

- 所有开发工作默认在 Git 仓库中进行。
- 若当前目录不是 Git 仓库，先提醒并协助用户执行仓库初始化、关联 GitHub 远程仓库或克隆现有仓库，再继续正式开发。
- 分支命名默认使用 `codex/` 前缀，例如：`codex/init-nextjs-app`、`codex/build-duty-overview-page`。
- 提交粒度保持小步快跑，每完成一个可验证的小任务就提交一次。
- 提交信息优先使用以下前缀：
  - `feat:`
  - `fix:`
  - `refactor:`
  - `test:`
  - `docs:`
  - `chore:`
- 未经用户明确要求，不要擅自执行破坏性 Git 操作，如 `reset --hard`、强制覆盖、删除分支。
- 若用户要求推送到 GitHub，优先先检查远程仓库配置、当前分支和工作区状态，再执行推送。

当前阶段的首要目标不是立即打通真实 Excel 导入，而是先完成以下能力：

- 首页值班摘要展示。
- 详情页联系人展示。

- 模拟数据驱动的完整前后端流程。

待真实 Excel 模板完全明确后，再继续实现正式导入逻辑。
